# ZeroTrust.sh — Architecture

This is the one deep-dive architecture document. It describes what is actually implemented and running today — not the original design vision, some of which (a Python ML worker, a CodeT5+ classifier) was scoped, partially built, and later removed as dead weight. The Docker-based PoE sandbox mentioned in that original design was rebuilt from scratch as `internal/poe` (opt-in `--verify-poc`, Phase 1 Java/Spring only) — see below. See `CLAUDE.md` for the short version and the current package map.

## Pipeline overview

```
Codebase
   │
   ├── Ingestion: Model Integrity Verifier + Differential Indexer
   │
   ├── Deterministic (pattern)         ├── Reasoning (semantic)
   │   OpenGrep + ast-grep + Joern taint        │   Heuristic Targeting (CPG-based)
   │   LLM Verifier (false-positive filter)     │   → Triage (lightweight LLM filter)
   │                                            │   → Analysis (full LLM reasoning)
   │
   └── Dedup + SSVC scoring → Postgres (findings, scan_runs) + patch suggestions
```

Both paths run concurrently; neither gates the other. A finding confirmed by both paths receives a +15pp confidence boost in Dedup.

## Ingestion

**Model Integrity Verifier (`internal/ingestion/miv`)** — verifies the LLM model binary (for local Ollama models) against a cosign/Sigstore Rekor-signed registry before the first LLM call. `WARN` on an unrecognized model, `BLOCK` on a hash mismatch. This gates LLM calls only — pattern matching and CPG taint analysis are unaffected by MIV status.

**Differential Indexer (`internal/ingestion/diffindex`)** — a Postgres content-hash cache keyed by project ID. On repeat scans, only changed files (plus their CPG neighbours, expanded via `internal/pipeline`'s module-scope helpers) are reprocessed.

## Deterministic — pattern detection

- **OpenGrep + ast-grep** (`internal/scanner/opengrep`) — rule-based pattern matching across the supported language set.
- **Joern CPG taint analysis** (`internal/cpg_engine`) — builds a Code Property Graph and runs inter-procedural taint queries (source → sink, with sanitizer detection).
- **LLM Verifier** — filters false positives from pattern findings. High-confidence rules bypass the verifier and go straight to Dedup.

## The CPG engine (`internal/cpg_engine`)

Wraps Joern's HTTP JSON API. Public surface: construct a `Client` with `New`, build the graph with `BuildCPG`/`LoadCPG`/`IncrementalPatch`, get a `Graph` with `Client.Graph()`. Everything else — the Joern DSL query builders, HTTP wire protocol, JSON parsing — is unexported.

Every Joern query lives in its own file under `internal/cpg_engine/queries/` — `.scala` for static queries, `.scala.tmpl` for parameterized ones (rendered via Go's `text/template`, `go:embed`'d at build time). This exists specifically so the Scala DSL can be read, diffed, and reasoned about as Scala, not decoded out of Go `fmt.Sprintf` string literals. Go's `{{.Field}}` substitution syntax is visually distinct from Scala's own `${expr}` interpolation, so there's no ambiguity about which language a given token belongs to.

Query-string unit tests (`queries_test.go`) confirm the generated text has the right shape (`.sortBy` before `.take`, expected JSON keys present, etc.) — they cannot confirm the Scala actually executes correctly against Joern. That's what `joern_integration_test.go` (build-tagged `integration`, requires a live Joern server) is for. Run it after touching anything under `queries/`. It's deliberately narrow now — it no longer builds a real CPG or runs a real taint query, since the Spring Boot fixture codebase those tests depended on wasn't checked into the repo (a real gap: unit tests alone don't confirm result *parsing* survives against non-trivial multi-node query output, which is exactly what `parseStdout` got wrong until an end-to-end smoke test caught it — see below).

**`parseStdout`'s triple-quote bug (fixed)** — Joern's REPL pretty-printer renders a query result as a Scala triple-quoted literal (`"""[{...}]"""`) whenever the underlying String contains literal newlines, which is the case for virtually every multi-node JSON-list query template in this codebase (they embed `\n` for readability). `parseStdout` only handled bare JSON and single-quoted+escaped Scala strings, so a triple-quoted result fell through unstripped, and every downstream `json.Unmarshal` call rejected it with `invalid character '"' after top-level value`. Caught during an end-to-end smoke test — every real query with actual matching nodes (`GetCallGraph`, `IngestCPGToDB`, non-empty `QueryNodesByFile`) failed silently, producing zero findings on a real codebase despite a confirmed, obvious vulnerability, while every unit test (which exercises `parseStdout` with hand-written single-line fixtures) passed. Fixed by detecting the `"""..."""` wrapper and stripping the 3 delimiter characters directly (the inner content is Scala's raw literal text, not JSON-escaped, so it must not be `json.Unmarshal`-decoded like the single-quote case). A regression test in `joern_test.go` reproduces the exact raw stdout captured from the live failure.

Incremental CPG patching: on repeat scans, the CPG is loaded from a serialized snapshot and patched with a depth-5 BFS update rooted at each changed function. Depth 5 is the taint-correctness bound from Li et al. (ICSE 2024) and Effendi et al. (SOAP/PLDI 2025) — see `docs/research-papers.md`. If a changed function has ≥50 callers (a "hub module"), the patch falls back to a full CPG rebuild.

`internal/pipeline/cpg_incremental.go` reads scope/depth config through `config.C` (the calibration-loaded global), not `config.New()` — an end-to-end smoke test caught 5 call sites still calling `config.New()` directly, which (a) silently ignores any `--calibration` override and (b) panics if the compile-time defaults themselves ever fail validation, which they did: `defaultConfig()` was missing `SummarizerMaxFunctionsPerBatch`, so its zero value failed the `gte=1` validation tag and crashed the whole process the first time a real scan reached a full CPG build. Both are fixed; `config.New()` should only ever be called where there's no loaded `config.C` to read (tests, mainly).

**Joern is on by default now.** `--joern-bin` used to default to `""`, which `startJoern` silently treated as "an externally managed Joern server is already running at the fixed `127.0.0.1:8080` URL" — with no warning if that assumption was wrong. Since almost nobody runs a Joern server that way, this meant CPG/taint analysis (Deterministic's Joern half, all of Reasoning, PoE route resolution) silently produced nothing on a default `zerotrust scan` invocation, with the scan still exiting cleanly. `--joern-bin` now defaults to `"joern"` (resolved via PATH, matching the Homebrew install convention used everywhere else); `main.go` also does a fail-fast-but-non-fatal `exec.LookPath` preflight that prints a loud stderr warning if the binary can't be found, and `startJoern` itself now logs (rather than silently returning) when `JoernBin == ""`.

**Warnings that used to vanish now reach `p.logger`.** A dozen-plus warning call sites across the pipeline (`portmgr.go`, `cpg_incremental.go`, `reasoning.go`, `pipeline.go`) constructed an `output.Event{Kind: output.EventLog}` and sent it only through the `events` channel — which `cmd/zerotrust/main.go` always passes as `nil` (there's no CLI progress renderer or SSE consumer in the current architecture). Since `output.Emit` on a nil channel is a documented, deliberate no-op, every one of those warnings (joern port conflicts, joern start failures, CPG build failures, commit-scan-state failures, opengrep/joern-taint errors) was completely invisible — not just off stderr, but absent from `build/zerotrust.log` too. All converted to `p.logger.Warn`/`Info`, which always reaches the JSON log file and additionally reaches stderr under `--verbose`.

## Reasoning — semantic detection

**Heuristic Targeting (`internal/semantic/targeting`)** — CPG-based surface selection: external-input nodes, auth-boundary nodes, call-graph depth, IDOR candidate detection (P-API/C-API taint model). Combined with Trivy CVE enrichment (`internal/semantic/enrichment`) — an exact CVE match auto-flags a surface and skips straight to Dedup.

**Contracts / DCC (`internal/semantic/contracts`)** — deterministic per-CWE invariant checks (the `Rulebook`) that classify each enriched surface as safe (drop), violation (escalate straight to the finding stream), or inconclusive (forward to Triage). The primary safe/violation signal is `enrichment.EnrichedSurface.Sanitized` — a structural fact from the CPG's own language-normalized taint taxonomy (`cpg_engine.TaintPath.Sanitized`), not a source-text keyword match, which is what keeps this working the same way across Java/Python/JS/Go rather than depending on per-language naming conventions. The rulebook's keyword-based `SafeNodes` list is kept only as a fallback for gaps the taxonomy doesn't yet cover. For the narrow remaining case — a confirmed, unsanitized-by-structure taint path where the keyword fallback also finds nothing — `contracts.NewWithEscalation` (used in production; plain `contracts.New()` stays fully deterministic, e.g. for tests) asks the LLM one scoped yes/no question ("does this specific function neutralize the tainted value") rather than assuming violation outright. Any error or non-affirmative answer defaults to violation — a scoped LLM call can *confirm* safety, it never overrides a structurally-confirmed danger signal by staying silent, the same asymmetric-trust rule used by Analysis's self-consistency check and by `internal/poe`.

**Triage (`internal/semantic/triage`)** — a lightweight LLM pass that scores each surface's exploitability on a continuous 0.0–1.0 scale (with an anchored calibration guide in the prompt). Surfaces below the confidence threshold are dropped; the rest escalate to full analysis. This remains the cheap, single-call gate ahead of the more expensive Analysis stage — Contracts getting sharper reduces how many surfaces reach it, it doesn't replace it.

**Crypto checks (`internal/semantic/crypto`)** — deterministic detection of weak/misused cryptographic primitives, run alongside the rest of Reasoning without needing an LLM call.

**Analysis (`internal/semantic/analysis`)** — the full LLM reasoning pass. One prompt strategy (no per-model tiers): Security Contract Layer + Control-Flow Predicate + AI-Failure-Profile evidence injected into the prompt, an externalized step-by-step reasoning scaffold, few-shot calibration examples, and a self-consistency re-check (a second, code-only LLM call) for any high-confidence exploitable verdict — this catches single-pass overconfidence before it becomes a reported finding.

When `Scanner.WithGraph` is set (wired in production via `pipeline.go`), the single-shot `Generate` call becomes a bounded tool-calling loop instead: the model gets up to `maxToolCalls` (4) round-trips to call read-only CPG queries — `get_callers`, `get_callees`, `get_neighbours_at_depth`, `query_nodes_by_file` (`internal/semantic/analysis/tools.go`) — before it must commit to a verdict. This is the "lite agentic" capability: real investigative depth ("is there an auth check within a couple of hops of this function") without DeepAudit's unbounded ReAct-style iteration budget. The cap is a hard stop, not a negotiable budget — if the model still wants another tool call when the cap is hit, one final tools-disabled call forces a text verdict, so per-surface cost is always computable in advance (at most 5 LLM calls: 4 tool rounds + 1 forced answer). Every tool call and its result is logged (`slog.Debug`, tagged `analysis: tool call`), so the investigation is auditable the same way a graph query is independently re-runnable — not "the model says it checked something." Without `WithGraph`, Scanner is unchanged from before: a single `Generate` call, no tools.

## LLM layer (`pkg/llm`)

Provider-agnostic. `llm.New(cfg)` returns a `Provider` (`Generate`, `Chat`, `Ping`, `ModelName`) regardless of backend. Ships with Ollama (default, local) and an OpenAI-compatible client (`--llm-provider openai`). Adding a third provider is a small new file plus one line in `llm.New`'s switch — the CLI surface (`--llm-provider`, `--model`, `--llm-base-url`, `--llm-api-key`) doesn't need to change.

**Retry/backoff (`pkg/llm/retry.go`)** — `llm.New` wraps whatever provider it constructs with `WithRetry` before returning it, so every caller (Analysis, Contracts escalation, Triage, PoE exploit-gen/grading, Patch) gets resilience against a transient Ollama/OpenAI hiccup for free, without needing to know about it. Up to 3 attempts (`DefaultRetryConfig`), exponential backoff with jitter, and the wait always respects the caller's context so cancellation isn't delayed behind a sleep. Two error classes are deliberately never retried: `ErrModelBlocked` (a MIV block must never be silently retried past) and `context.Canceled`/`context.DeadlineExceeded` (the caller's own cancellation takes priority). `Ping` passes through unwrapped — it's a one-shot startup reachability check where the caller wants an immediate fail-fast signal, not a delayed one. No circuit breaker: a breaker protects a long-lived server from a persistently failing dependency across many future requests, but a scan is a single short-lived process — if the backend is down for the whole run, every retry budget exhausts and every caller sees the same error regardless, so there's no "next request" for a breaker to protect. `SetProviderMIVBlocked` unwraps the retry wrapper before its `*ollamaClient` type assertion — a real bug caught while wiring this in: without the unwrap, MIV blocking would have silently stopped working the moment every provider started coming back wrapped.

**Tool/function calling** — `Options.Tools` (`[]ToolDef`, name + description + JSON-Schema parameters) and `Message.ToolCalls` let a `Chat` call request one or more tool invocations instead of a final answer; the caller dispatches them and feeds results back as `RoleTool` messages. Both backends speak this natively rather than through a lossy shared abstraction: OpenAI's SDK (`github.com/openai/openai-go`) maps directly onto its own `Tools`/`ToolCalls` fields, and Ollama's `/api/chat` gets its own wire-shape conversion (`toOllamaMessage`/`fromOllamaMessage` in `pkg/llm/ollama.go`) since Ollama represents tool-call arguments as a JSON object on the wire where OpenAI (and this package's public `ToolCall.Arguments`) use a JSON-encoded string — the conversion re-encodes between the two so callers only ever see the OpenAI-style string convention regardless of backend. This is the shared building block for both Contracts' scoped escalation and any future bounded tool-calling loop in Analysis; Contracts' current escalation call doesn't need tools itself (it only uses `Generate`), but the same `Provider` now supports both.

## Dedup + SSVC scoring (`internal/dedup`)

Cascaded dedup gates (exact match → near-miss embedding → AST edit distance), then SSVC-inspired confidence scoring sourced from CVSS, EPSS, and CISA KEV data where available. Cross-path confirmation (both Deterministic and Reasoning independently flag the same finding) adds a +15pp confidence boost.

## Sandboxed PoC verification (`internal/poe`, opt-in via `--verify-poc --poe-artifact <path>`)

Everything above is LLM-verdict + static-evidence based — no finding is ever actually exploited. `--verify-poc` closes part of that gap: it boots a real, already-built artifact in a sandboxed Docker container, fires an LLM-crafted HTTP request at the route that reaches the finding's sink, and grades the response. This runs once per scan (package once, boot once, fire N requests, tear down once), after Dedup and before Patch, and only against BLOCK/HIGH findings.

The container's network is a plain (non-`--internal`) bridge network, not a network-isolated one — a deliberate correction after live testing against Docker Desktop showed `--internal` networks silently break the host's own `-p` port-publish path (no NAT, so every exploit request would time out) and that connecting by bridge IP instead doesn't work either on Docker Desktop for Mac/Windows (its host-to-bridge route goes through a Linux VM that also doesn't forward into internal networks). The accepted trade-off: the sandboxed app can reach the internet, since PoE targets a developer-supplied artifact for a scan the developer requested, not a payload that needs containing from a hostile actor. If stronger egress isolation is ever required, the fix is a small proxy sidecar container that joins both an internal network and a published host port — not attempted here since it doesn't change today's actual risk profile.

**Grey-box, not build-from-source.** The caller supplies the exact artifact to test — a jar, a bundled JS file, a Python script, or a native binary (`--poe-artifact`) — and PoE never compiles anything. This is a deliberate trade against an earlier build-from-source design: skipping the compile step turns "support N languages" into "N runtime-only Dockerfiles" instead of "N build-tool detectors + N multi-stage compile pipelines," and testing the artifact that's actually going to run is arguably more representative than testing whatever the sandbox happens to compile. The trade-off, accepted for now and not yet mitigated: nothing verifies the supplied artifact was actually built from the source revision being scanned. A stale or mismatched artifact produces misleading results — assumed correct today, drift detection is a deliberate follow-up.

Flow: `poe.Verifier.Run` — detect the artifact's runtime from its extension/executable bit (`internal/poe/build.go`) → stage it under a fixed in-container filename and `docker build` a static, non-templated runtime-only Dockerfile (`internal/poe/dockerfiles/{java,python,node,go}-runtime.Dockerfile` — COPY + run, no compile stage, reusing the same unprivileged `nobody` user as the original `docker/sandbox/Dockerfile`) → boot on a plain bridge network (see above for why not `--internal`) with `--read-only --memory 512m --cpus 1.0 --user 65534:65534` and a seccomp profile → health-poll the mapped port → per eligible finding: resolve an HTTP route, ask the LLM for an attack payload at that route, fire it, grade the response → tear down. The seccomp profile is `internal/poe/seccomp-profile.json`, embedded into the binary via `go:embed` and staged to a temp file at boot time — not the cwd-relative `docker/sandbox/seccomp-profile.json` path an earlier version used, which silently skipped the profile (with the container still running, just less hardened) whenever `zerotrust` was invoked from anywhere other than the repo root. Confirmed via a live boot that the staged profile actually reaches `docker run`'s `--security-opt`, not just that the file exists.

**Route resolution (`internal/poe/route.go` + one `route_<lang>.go` per language)** walks the CPG call graph backward (`Graph.GetCallers`, depth-bounded at 5, matching the same taint-path correctness bound used elsewhere) from the finding's enclosing method looking for a recognized HTTP entry point, dispatched by `finding.LangFromPath` to a per-language `routeResolver`:
- **Java** (`route_java.go`) — Spring mapping annotations (`@GetMapping`/`@RequestMapping`/etc.) directly above the method, plus a class-level `@RequestMapping` prefix.
- **Python** (`route_python.go`) — FastAPI (`@app.get("/path")`) and Flask (`@app.route("/path", methods=[...])`) decorators, same annotation-window approach as Java.
- **JavaScript/TypeScript** (`route_node.go`) — Express route registration (`app.get('/path', handler)`); TypeScript findings resolve here too since the artifact is expected to be the already-transpiled/bundled JS output. Unlike Java/Python, the registration call doesn't sit next to the handler's declaration, so this scans the whole file for a call whose handler argument matches the function name — same-file only, a router file that imports a handler defined elsewhere won't resolve.
- **Go** (`route_go.go`) — `net/http`'s `HandleFunc` (verb determined inside the handler, defaults to GET at the registration site) and Gin/chi/Echo-style `r.GET(...)`/`r.POST(...)` calls, same whole-file scan as JS.

None of this is a CPG query — `cpg_engine` has no Annotation/Decorator node type today, and four languages' routing conventions are too varied to express as one shared Scala query; each resolver is a plain-text regex scan of the source. A finding with no traceable HTTP entry point gets `finding.PoEInconclusive`, never a guessed route.

**Grading** runs a cheap deterministic pre-check first (a 5xx response containing a known error-leak substring like `SQLException` is an unambiguous confirm) and only falls back to a narrow LLM classifier call (mirrors `internal/semantic/triage`'s shape: low temperature, small token budget) for the ambiguous remainder.

Scope, deliberate:
- **No external service provisioning.** Artifacts that need a live DB/Redis/etc. to boot cleanly fall back to `finding.PoEInconclusive` rather than a faked dependency.
- **Eligibility is declarative config, not a hardcoded gate.** `partitionEligible` reads `config.C.PoEMinSeverity` (minimum severity worth verifying, default `"HIGH"`) and `config.C.PoESupportedLanguages` (default `["java","python","javascript","typescript","go"]`) from `calibration.json`'s `poe_min_severity`/`poe_supported_languages` keys, so the escalation policy is inspectable/changeable without a code change. An unrecognized severity name falls back safely to the compile-time default rather than silently misreading as `"BLOCK"`.
- **A failed or inconclusive attempt never lowers a finding's confidence** — only `finding.PoESuccess` applies a boost (`config.BoostPoEConfirmed`, re-deriving severity via `finding.SeverityFromConfidence`). A sandboxed miss doesn't prove the real deployment is safe (different env vars, different DB state), so the asymmetric-trust posture already used for the self-consistency check in `internal/semantic/analysis` applies here too.
- **No artifact-freshness check.** See the grey-box trade-off above — a real gap, not an oversight.

## Persistence (`pkg/postgres`) — the database is the product

Scored findings are persisted to Postgres, not rendered to a report. `internal/pipeline`'s `persistFindings`/`finalize` upsert each finding into the `findings` table and finalize the `scan_runs` row — plus, for each finding, its SSVC dimensions into `ssvc_scores` (when at least one dimension is set). PoE verification runs after `StartScanProcess` returns (`cmd/zerotrust/main.go`, since it needs the caller-supplied `--poe-artifact` path the pipeline itself has no reason to know about), so by the time it produces a `finding.PoEResult`, `persistFindings` has already written that finding's row — with an empty `poe_results` entry. `Pipeline.PersistPoEResults` is the second, explicit write that lands the real verdict (and any `PoESuccess` confidence/severity boost) into `findings`/`poe_results` after verification completes; skipping it would have silently defeated the point of persisting PoE results at all. There is no HTML report, no JSON report, and no CLI progress renderer — `cmd/zerotrust/main.go` calls `p.StartScanProcess(ctx, nil)` and the only output is structured logging (`build/zerotrust.log`, stderr under `--verbose`) plus the process exit code (documented CI contract: `0` clean, `1` one or more BLOCK/HIGH findings).

`pkg/postgres` uses GORM (`gorm.io/gorm` + `gorm.io/driver/postgres`) for the CRUD-shaped tables (`projects`, `scan_runs`, `findings`, `ssvc_scores`, `poe_results`, `cpg_cache`, `scan_state`, `suppressions`, plus `work_items`/`pending_findings` — schema exists, not yet wired to a writer) and a raw `pgx`/`pgxpool` connection for the CPG node/edge bulk path (`IngestNodeBatch`/`IngestEdgeBatch` via COPY, and the recursive-CTE `GetNeighboursAtDepth` BFS query) — that's the actual high-volume write path per scan, and COPY is materially faster there than row-by-row ORM inserts. This replaced an earlier SQLite-backed `pkg/sqlite`; Postgres's MVCC handles concurrent reads+writes over one pool natively, removing the single-writer WAL workaround SQLite needed, and gives a real query engine for future RAG-style workloads.

**Deprecated, not deleted:** `internal/report` (HTML report) and `internal/output`'s CLI renderer still exist as packages but are unwired from the pipeline — kept in place for when a client UI gets built, rather than reactivated piecemeal. `internal/patch` is still active: `--patch` generates unified-diff patch suggestions via the LLM, validated with `go-gitdiff` (a patch that fails to apply is marked `malformed`, never silently presented as good) — the result now lands in the `findings.patch`/`patch_status` columns instead of a report tab.

## What's deliberately not here

- **No Docker orchestration in the binary.** It was removed entirely — users who want a container build/run one themselves.
- **No Python ML worker, no CodeT5+ classifier, no XGrammar, no LangGraph.** These were part of the original design and got built, but the actual classifier tier that survived is a pure-Go heuristic (`taintGateClassify` in `internal/pipeline/reasoning.go`). The Python code was disconnected from the pipeline and deleted rather than left as dead weight.
- **No token budget cap / "Token Budget Controller."** It was already a no-op before removal — nothing capped or suppressed findings for cost reasons in practice, so the config surface (`--token-cap`, calibration budget weights) was removed to match reality.
- **No sandbox verification beyond Phase 1's Java/Spring scope.** `internal/poe` (`--verify-poc`) covers Maven/Gradle Spring projects with no external service dependency; everything outside that (other languages, DB-dependent boot, non-HTTP entry points) still falls back to LLM-verdict + static-evidence only. See the "Known gaps" section in `CLAUDE.md`.
- **No HTML/JSON report, no CLI progress renderer.** `internal/report` and `internal/output`'s renderer are unwired, not deleted (see "Persistence" above) — findings live in Postgres and there's no UI on top of it yet.
- **No SQLite anywhere.** The prior `pkg/sqlite` state cache was fully replaced by `pkg/postgres`; there is no embedded/file-based database left in this codebase.
