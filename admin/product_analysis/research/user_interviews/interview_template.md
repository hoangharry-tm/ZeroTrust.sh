# Developer Interview Template — ZeroTrust.sh Value Proposition Validation

**Purpose:** Validate or invalidate the core assumptions behind ZeroTrust.sh, specifically: (1) developers using AI coding agents experience real security concerns, (2) local/offline scanning is preferable to cloud alternatives, and (3) AI-specific threat vectors (slopsquatting, prompt injection) are perceived as meaningful risks.

**Interview format:** 30–45 minutes, semi-structured. Ask all numbered questions; follow up naturally. Take verbatim notes on Q4, Q6, Q8, Q9, Q10 — these are the highest-signal questions.

**Interviewer note:** Do not pitch ZeroTrust.sh during the interview. Describe it only in Q9 and only after capturing the participant's unprompted views. Avoid leading language.

## Interview Questions

### Section A — Current Security Practices

**Q1: Walk me through how you review code before it goes to production. What does your security review process look like today?**

*What to listen for: whether any security review exists at all; tools used; manual vs automated; how thorough they believe it is.*

---

**Q2: What security tools do you currently use in your development workflow — linters, SAST scanners, dependency checkers, anything like that?**

*Probe:* "Are any of these local or offline tools? Any cloud-based ones?"
*What to listen for: tool names (Semgrep, Snyk, CodeRabbit, Dependabot); awareness of the local vs cloud distinction; friction with current tools.*

---

**Q3: Have you ever found a real security vulnerability in your own code — either before or after release? What happened?**

*What to listen for: emotional response to security incidents; severity; whether it came from manual review or a tool; whether it changed their behavior.*

---

### Section B — AI Coding Agent Experience

**Q4: How much of the code you write today is generated or significantly modified by an AI coding agent (Cursor, GitHub Copilot, Cline, Aider, or similar)?**

*Probe:* "Has that proportion changed in the last 12 months?"
*What to listen for: honest percentage estimate; trajectory of AI adoption; whether they distinguish AI-generated code from their own.*

---

**Q5: When you accept AI-generated code, do you review it differently than code you write yourself? Why or why not?**

*What to listen for: whether they trust AI-generated code implicitly; whether they apply additional scrutiny; specific concerns or lack thereof.*

---

**Q6: Have you ever encountered a problem — a bug, a security issue, or just something that felt wrong — in code that was generated by an AI tool? Can you describe what happened?**

*What to listen for: concrete examples are gold; note whether the issue was security-related or purely functional; how it was discovered; whether it was preventable.*

---

### Section C — Privacy and Tool Adoption

**Q7: Some developer security tools (like Snyk and CodeRabbit) work by sending your source code to their cloud servers for analysis. Does that concern you? Why or why not?**

*What to listen for: strong reactions (positive or negative); whether they have company policy constraints; whether privacy concern is theoretical or has affected actual tool choices.*

---

**Q8: If you had to choose between a tool that runs entirely on your machine (takes 10–30 seconds, requires downloading a 4 GB model) versus a cloud tool that returns results in 2 seconds but processes your code on a remote server — which would you choose, and what would drive that decision?**

*What to listen for: the tradeoff they actually make vs what they say they value; whether speed, privacy, or cost dominates; whether 4 GB download is a dealbreaker.*

---

### Section D — ZeroTrust.sh Concept Validation

**Q9: I want to describe a tool concept and get your honest reaction. [Describe ZeroTrust.sh: local CLI scanner, no code egress, targets AI-generated code specifically, detects things like fake package names and injected instructions in comments, outputs a patch-ready HTML report.] What's your immediate reaction? Would you use this? What concerns do you have?**

*Probe:* "What would need to be true for you to actually install and run it on a real project?"
*What to listen for: enthusiasm vs skepticism; specific objections; whether the AI-specific angle resonates; friction points in the described workflow.*

---

**Q10: If this tool existed today and was free and open-source, what would stop you from adopting it? And what would make it a must-have rather than a nice-to-have?**

*What to listen for: adoption blockers (setup friction, trust, performance, compatibility); the "must-have" trigger often reveals the real value driver; note exact wording.*
