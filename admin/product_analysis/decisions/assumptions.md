# Assumptions Register

All assumptions currently baked into the ZeroTrust.sh analysis and proposals are tracked here. Each assumption must be explicitly validated before the final approach selection and implementation begin.

**Status Legend:** `Unvalidated` — not yet tested | `Partially Validated` — some evidence exists but not conclusive | `Validated` — confirmed with evidence | `Invalidated` — proven false, requires rework

---

## Assumptions Table

|---|---|---|---|---|
| A-01 | Developers care enough about source code privacy to accept local compute overhead (slower scan times, VRAM usage) compared to instant cloud-based alternatives | Market | Unvalidated | Run 5+ structured developer interviews; ask directly about privacy vs convenience tradeoff; review existing privacy-tool adoption data (e.g., local LLM tool usage stats) |
| A-02 | Target machines (developer laptops) have at least 8 GB RAM available for concurrent LLM inference alongside a running IDE and development tools | Technical | Unvalidated | Survey developer hardware profiles via interview; cross-reference with GitHub Copilot system requirements and Ollama minimum requirements documentation |
| A-03 | `qwen2.5-coder:7b` (or equivalent 7B-parameter quantized model) is accurate enough for security vulnerability classification — not just code completion benchmarks | Technical | Unvalidated | Design and run a security-domain benchmark: feed known-vulnerable code snippets (OWASP top 10, CWE-25) to the model and measure precision/recall vs ground truth |
| A-04 | Tree-sitter grammars exist and are sufficiently complete for the MVP target languages: **Python, TypeScript, JavaScript, Go, Rust, Java, C, C++, C#, Kotlin, Swift, PHP, Ruby, Bash, Scala** | Technical | Partially Validated | Tree-sitter has grammars for all listed languages; C/C++ completeness requires special attention (preprocessor macros, `#include` resolution fall outside Tree-sitter's scope and require a separate handling strategy) |
| A-13 | C and C++ can be meaningfully scanned at the AST level despite preprocessor macros (`#define`, `#include`) that Tree-sitter does not expand — scanning will operate on post-preprocessor or raw source depending on the approach selected | Technical | Unvalidated | Prototype Tree-sitter C/C++ grammar against real AI-generated C++ code; identify which vulnerability classes are detectable without macro expansion vs which require a full preprocessor pass (e.g., via `clang -E`) |
| A-14 | Slopsquatting detection is viable for all MVP languages — each has a queryable package registry: PyPI (Python), npm (JS/TS), Maven Central (Java/Kotlin), crates.io (Rust), NuGet (C#), Swift Package Index (Swift), Packagist (PHP), RubyGems (Ruby). **C and C++ have no authoritative central registry** (vcpkg and Conan exist but are fragmented) — slopsquatting detection for C/C++ requires a different strategy or is out of scope for V1 | Technical | Unvalidated | Verify API availability and rate limits for each registry; design a C/C++ fallback strategy (e.g., scan CMakeLists.txt / vcpkg.json for known-bad package names rather than live registry lookup) |
| A-05 | Ollama provides a stable and production-appropriate local HTTP API for model inference, suitable for integration into a CLI tool without significant reliability overhead | Technical | Partially Validated | Ollama is widely used in production contexts; need to test API stability under concurrent requests, cold-start latency, and error recovery in a pipeline context |
| A-06 | AI coding agent adoption (Cursor, Cline, Aider, Copilot Workspace) will continue growing, sustaining and expanding the addressable market for AI-code security tooling | Market | Unvalidated | Monitor Cursor/GitHub Copilot user growth metrics; track enterprise adoption announcements; review developer survey trends (Stack Overflow, JetBrains) over 6-month horizon |
| A-07 | Package hallucination (slopsquatting) is detectable via real-time registry API lookup (PyPI, npm, crates.io) without violating the offline-first constraint or hitting rate limits at scale | Technical | Unvalidated | Prototype a registry lookup module; measure API response times and rate limit thresholds at scan-time volume; evaluate offline fallback options (cached known-package lists) |
| A-08 | The primary ICP (ideal customer profile) is a solo developer actively using AI coding agents who has experienced or fears AI-introduced security bugs — not a dedicated security team | Market | Unvalidated | Validate via developer interviews; check if security teams already have SAST tooling that covers their needs; test willingness-to-pay signal in both segments |
| A-09 | An open-source core model (free CLI tool + optional paid enterprise dashboard) is viable and will drive community-sourced rule contributions that sustain the rule engine quality | Market | Unvalidated | Study comparables: Semgrep OSS vs Semgrep Pro adoption ratio; Trufflehog community contribution velocity; assess whether a developer security tool community will form around ZeroTrust.sh |
| A-10 | Indirect prompt injection in code comments and docstrings is a real, demonstrated threat that developers will perceive as worth scanning for — not a theoretical/academic concern | Market | Partially Validated | Document real-world examples of prompt injection via code artifacts; measure developer awareness in interviews; link to CVE or public incident reports if available |
| A-11 | A single compiled binary distribution (for Go/Rust implementations) is the preferred delivery mechanism for developer CLI tools in 2026 and will minimize friction in the install-to-first-scan flow | Technical | Unvalidated | Survey developer install preferences; compare friction of `brew install`, `curl \| sh`, and pip install patterns; test install-to-first-scan time on a fresh machine |
| A-12 | The LangGraph multi-agent approach (Approach 3) can be packaged into a Docker container that runs reliably on developer laptops without requiring Docker Desktop licensing workarounds | Technical | Unvalidated | Prototype Docker sandbox execution on macOS (Docker Desktop) and Linux; measure startup overhead; investigate Colima/Podman as license-free alternatives |

| A-15 | ZeroTrust.sh's primary user is the **developer of the application** — not a security team, not a pentester, not a QA engineer. Pentesters need Strix/XBOW/RidgeGen (deployed app testing). QA needs UAT with human-in-the-loop for business logic. Developers need a local tool that audits what their AI coding agent just wrote | Market | Partially Validated | Confirmed by user persona analysis (Personas 1 and 2); further validation via developer interviews required. The key marker: does the user run this tool in their local terminal after an AI agent session, not as a scheduled security engagement? |
| A-16 | The developer-workflow position (pre-commit, post-AI-agent-session) is meaningfully distinct from CI/CD scanning (post-commit) and pentest (post-deployment) in the developer's mental model — they reach for different tools at each stage | Market | Unvalidated | Validate via developer interviews: ask "when would you run ZeroTrust.sh?" and map their answer to the SDLC stage. If they describe post-commit CI/CD use, the tool competes with Semgrep. If they describe post-deployment use, it competes with pentest tools. The hypothesis is they describe a pre-commit, post-AI-session use. |
| A-17 | The two-path architecture (Path A: fast AST rules + Path B: independent LLM logic scan) is feasible to implement as a unified developer CLI tool that completes a full scan in under 5 minutes on a typical developer machine (16 GB RAM, Apple M-series or equivalent) | Technical | Unvalidated | Prototype Path B heuristic targeting + LLM surface scan against a 10K-LOC test codebase; measure wall-clock time; validate against 5-minute developer patience threshold |
| A-18 | The Code Vulnerability Classifier (UniXcoder-Base-Nine) F1=94.73% claim is based on the BigVul dataset (C/C++ functions, pre-2020 CVEs). This benchmark does not reflect the actual target input: AI-generated Python, TypeScript, Java, and Go code from 2023–2026 coding agents. The F1 on AI-generated code from tools like Cursor, Cline, and Copilot Workspace is unknown and may be materially lower — AI-generated code has different structural patterns (more boilerplate, different naming conventions, higher use of LLM-idiomatic API chains) than the human-written C/C++ in BigVul | Technical | Unvalidated | Design a benchmark specifically targeting AI-generated code: (1) generate 200–300 code samples across Python/TS/Java using Cursor, Cline, and Copilot Workspace with intentional vulnerability introduction; (2) run UniXcoder-Base-Nine against ground-truth labels; (3) measure precision, recall, F1 on this corpus; (4) if F1 drops below 0.85, evaluate GraphCodeBERT or VulGNN (lightweight GNN, LLM-parity at 100x smaller size per Zhu et al. 2026) as replacements. This benchmark must be completed before publishing detection accuracy claims. |
