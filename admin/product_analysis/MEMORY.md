## Section 2 — Three Approaches Being Evaluated

All three approaches solve the same core problem (AI-agent code vulnerability detection) but with fundamentally different tradeoffs in accuracy, speed, resource requirements, and implementation complexity.

| Approach | Name | Summary |
|---|---|---|
| **1** | Pure AST Static Analysis | Tree-sitter parses code into AST; YAML-defined rules match structural vulnerability patterns. No LLM required. Fast, deterministic, low resource footprint, but limited to pattern-match accuracy. |
| **2** | Hybrid AST + Local LLM | Tree-sitter pre-filter (high recall) feeds suspicious snippets to a locally-hosted quantized LLM (via Ollama) for semantic verification and false-positive reduction. Patch suggestions generated by LLM. Balanced accuracy/resource tradeoff. |
| **3** | Multi-Agent Sandbox | LangGraph-orchestrated agents perform multi-pass analysis. A Docker-isolated sandbox executes payloads to verify exploitability. Highest accuracy and context-awareness, but heaviest resource requirements and most complex architecture. |

Detailed documents: `specs/proposals/approach_1_ast_sast.md`, `specs/proposals/approach_2_hybrid_llm.md`, `specs/proposals/approach_3_multi_agent.md`

## Section 5 — Key Technical Context

### AI-Specific Threat Vectors Targeted

ZeroTrust.sh is differentiated by targeting threat classes that general-purpose SAST tools do not detect:

| Threat | Description |
|---|---|
| **Package hallucination (slopsquatting)** | AI agents generate `import` statements referencing non-existent packages; attackers pre-register those names with malicious payloads |
| **Indirect prompt injection** | Malicious instructions embedded in code comments, docstrings, or string literals designed to hijack future AI agent sessions reading the file |
| **Vibe coding security deficit** | Structural patterns indicating security controls were removed or weakened during AI-generated refactoring (e.g., authentication bypass, input validation removal) |
| **Safety gate bypass** | AI-generated code that circumvents existing security middleware, sanitization functions, or access control layers |

### Privacy Model

All analysis runs fully local and offline. No source code, file paths, or metadata are transmitted to any external service. This is a hard architectural constraint, not a feature toggle.

### Competitor Landscape

| Tool | Approach | AI-Specific Detection | Privacy Model |
|---|---|---|---|
| **Semgrep** | Local AST pattern matching | No | Local |
| **Snyk** | Cloud SAST | No | Cloud (source code uploaded) |
| **CodeRabbit** | Cloud PR review with GPT | Partial (code quality) | Cloud (source code uploaded) |
| **Trufflehog** | Local secrets scanning | No | Local |
| **Bearer** | Local SAST + data flow | No | Local |
| **ZeroTrust.sh** | Local hybrid analysis | Yes (4 AI-specific vectors) | Local (hard constraint) |

**ZeroTrust.sh's unique angle:** The only local scanner with purpose-built AI-specific threat detection.
